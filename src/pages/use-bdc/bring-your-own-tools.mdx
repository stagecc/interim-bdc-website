## Overview

BDC enables researchers to bring their own tools, scripts, and workflows (also known as “BYOT”) into a secure, cloud-based environment for scalable, reproducible analysis. In BDC, tools can take several forms; these components are combined to create scalable, shareable analyses in BDC's analysis platform, *BDC-Seven Bridges*. 

* **Tool**: Single command or script (R, Python, Bash) that performs one task.  
* **App**: Tool or full workflow packaged for use.  
* **Workflow**: Multi-step pipeline connecting tools into a reproducible analysis.

<InfoCard>
### Important Guidance for Regulated and AI-Based Tools

In some cases, researchers or teams may be developing tools for more advanced or regulated use cases. If you are developing tools for clinical or regulatory use (such as Software as a Medical Device), contact the BDC team early, particularly before pursuing FDA approval, to ensure alignment with BDC requirements.

Additionally, while AI-assisted tools can accelerate development, they may introduce risks related to security, reproducibility, and transparency. Tools brought into BDC must meet requirements for controlled-access data and reproducibility, and will need to be validated prior to use.

[Submit a request through our Help Desk](/help-and-support/contact-us/) to get started. Our team can help assess your approach and guide next steps.
</InfoCard>

### Want to BYOT to BDC?

Whether you've developed a custom tool or pipeline, packaged a tool in Docker, or are exploring how your analysis could run on BDC, we can help you evaluate your approach. Our team can help determine feasibility, guide implementation, and avoid common pitfalls. Early conversations save time and prevent rework.

#### Need help getting started?

If you are exploring how to bring a tool or workflow to BDC and are unsure where to begin, our team can help. You should reach out if you:

* Are planning to publish a tool in the Apps Gallery.  
* Are working with custom or containerized tools and need help integrating.  
* Have never created a CWL app in *BDC-Seven Bridges* before.  
* Are unsure how to move from a script or analysis to a scalable workflow. 

If you are considering bringing a tool to BDC, submit a request through our help desk and select “Tool or Workflow Integration.”

### You May Benefit from BYOT If…

You may benefit from beginning a tool to BDC if:

* You have a script you want to run across multiple files or reuse over time.  
* Your analysis is expected to take several hours.  
* You have a multi-step workflow you want to formalize and scale.   
* Your workflow has many dependencies and would benefit from containerization.  
* You want to reuse or integrate an existing Dockerized tool.  
* You want application versioning.

### Important Considerations

Not all tools are immediately compatible. Before getting started, consider:

* Data compliance and security requirements: Tools must meet requirements for controlled-access data and secure environments.  
* Tools that depend on proprietary software (for example, MatLab) or have licensing restrictions are not supported.  
* Tools or workflows that rely on interactive inputs or provide interactive visualizations are not typically supported through standard, self-service workflows. These use cases may be supported through dedicated, funded projects within BDC.  
* Tools must be containerizable and describable in supported workflow languages. Thus tools requiring direct hardware access, always-on services, and tools with no defined input are not supported.  
* Multi-container architectures are not supported.  
* Tools with unavailable or non-portable dependencies are not supported.

If you're unsure, contact us so we can help assess your approach.

## Before You Start

Bringing tools into BDC typically requires:

* Familiarity with command-line tools or scripts (R, Python, Bash)  
* An understanding of inputs, outputs, and dependencies necessary for your tool(s)  
* Experience with Docker or willingness to learn

### How It Works

At a high level, bringing a tool into BDC typically involves:

1. Understanding what you're bringing  
   * Determine whether you are bringing in a tool (single script) or a workflow (multi-step pipeline).  
2. Define your too'’s specifications  
   * Clearly define:  
     * Inputs (files required by your application)  
     * Parameters (any variables required by your application)  
     * Expected outputs   
     * Dependencies (libraries, packages)  
     * Resource requirements (CPU, memory, runtime expectations)  
   * Understand if it should be Dockerized   
     * *Note: This will be based on details such as the number of dependencies, install vs load times, and your familiarity with Docker.*  
3. Testing your tool  
   * Run the tool on sample data  
   * Validate outputs  
   * Iterate and refine before scaling or combining into workflows  
4. Creating and register your tool  
   * Use the Tool Creator to create command line tools  
   * Reference an open source Docker image or provide your own (for example, via DockerHub or another registry)  
   * If the Docker image requires credentials/ is not publicly accessible, register the tool using the [BioData Catalyst Image Registry](https://sb-biodatacatalyst.readme.io/docs/the-image-registry)  
5. Optional: Creating a workflow  
   * Combine multiple tools or scripts into a workflow  
   * Configure how tools connect and pass outputs  
   * Build  workflows using:  
     * Seven Bridges GUI-based Workflow Editor  
     * CWL definitions  
6. Sharing and scaling  
   * Share with collaborators within your project   
   * Potentially publish in the Public Apps Gallery (requires coordination with BDC3 and Velsera)  
   * Run analysis across your full dataset  
   * Explore batch processing through use of parallel executions
